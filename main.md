### ➡️ Markdown 사용을 추천하는 이유
- github 에서 쓰고 있음
- 우리 팀에서 쓰고 있음
- 생각을 빠르고 간단하게 구조화 + 서식화 할 수 있음
  ```mermaid
    graph TD;

  ```

- 이번 발표 자료도 이걸로 만들어보니 좋은듯 ~~(*사실 나도 많이 안써봤...*)~~

### ➡️ 매뉴얼 작성할 때 팁
- 갑자기 장애가 나서 누군가 나 대신 이 업무를 처리해야 하는데,
- 지금 당장 이 일을 해결해 줄 사람이 우리 사장님 밖에 없다고 가정한 뒤,
- 사장님이 내 자리에 오셔서 이 매뉴얼만 보고 순서대로 따라 해도 처리가 가능한가?
  - 그 사장님은 사실 모든 것을 다 잊어먹은 '미래의 나'
  - 즉, 상세한 매뉴얼은 현재의 내가 미래의 나에게 주는 선물
  > 언젠간 과거의 나에게 고마워 할 날이 온다.
- 중요한 **키워드**는 다양한 방식으로 **문서 제목과 내용에 포함** 시키고 태그 등으로도 표시해 놓자.
  - 키워드들은 **검색**에 활용 된다.
  - 반드시 미래의 내가 고마워 할 날이 또 온다.
- 거창한 양식, 형식, 필요 없고 업무에 대한 나의 기록물이라고 생각하자.

### ➡️ 매뉴얼(=기록물) 모아놓을 때 팁
- **분류**에 별로 소질이 없는 타입이라면, 차라리 **분실**만 되지 않도록 잘 모아놓자.
  - 키워드를 잘 써놓고, 검색을 잘 할 수 있다면 분류가 잘 안된 것은 커버 가능
  - 생성연도에 따른 분류 정도는 있으면 좋음. 해당 연도에 뭘 했는지 알 수 있으니 나중에 자기평가 할 때 활용 가능.
- 날짜와 시간을 잘 기록해두면 꽤 쓸모가 있음
- 사소한 것도 기록해 놓으면 분명 도움이 될 때가 있음
